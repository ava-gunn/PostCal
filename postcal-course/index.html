The Big Picture

What PostCal does and the user journey traced through code.

What is PostCal?

You see a concert flyer on Instagram. You want it in your calendar. Normally you would screenshot it, squint at the date, and manually type everything into a new event.

PostCal eliminates that friction. Share the Instagram post, review the pre-filled details, and save — done.

1
Share

Tap the share button on any Instagram event post and pick PostCal from the share intent list.

2
Review

PostCal reads the image and caption, then shows you a form with the event name, date, time, and venue already filled in.

3
Save

One tap writes the event straight to your device calendar. No account, no cloud, no waiting.

💡
Think of it this way

PostCal is like a personal assistant who reads event flyers for you and fills out the calendar entry — you just approve it.

The Tech Behind It

PostCal is built with a focused stack of libraries, each handling one job. Every piece runs on the device — nothing leaves the phone.

⚛️
React Native / Expo

The app framework. Write once in JavaScript, run natively on iOS and Android.

🔷
TypeScript

The language. Adds type safety on top of JavaScript so bugs surface at build time, not in production.

👁️
expo-text-extractor

The OCR engine. Uses on-device vision APIs to read text from images without sending anything to the cloud.

📆
chrono-node

The date parser. Understands natural language dates like “next Friday at 8pm” and converts them to real timestamps.

📅
expo-calendar

The calendar writer. An SDK module that writes events directly to the native iOS or Android calendar.

🧠
React Context

The state manager. Shares extracted event data across every screen without prop-drilling.

🔒
Why everything runs on-device

Privacy: your Instagram data never leaves your phone. Cost: no servers to pay for. Reliability: works offline, in airplane mode, with zero latency. This is the core design principle behind every technical choice in PostCal.

The Extraction Pipeline

Deep dive into how PostCal reads and understands shared content.

The Assembly Line

The extraction pipeline is like a team of translators working on a foreign document. Each specialist handles one part — dates, names, addresses — and their work gets combined into a complete translation.

No single stage does everything. Instead, seven focused steps each do one job well, then pass their results forward.

1
parseShareIntent(shareIntent)

Extracts the raw text, imageUri, and mimeType from the incoming share data.

2
extractTextFromImage(imageUri)

Runs on-device OCR via expo-text-extractor (ML Kit on Android, Apple Vision on iOS). Returns joined text blocks or null.

3
mergeExtractionSources(metadataText, ocrText)

Combines caption text and OCR text with a separator \n---\n so downstream parsers can distinguish the two sources.

4
parseDateTime(rawText)

Uses chrono-node to parse natural language dates. Returns { date: "YYYY-MM-DD", time: "HH:MM" }.

5
parseEventName(rawText)

Finds the first non-URL line of text. Prioritizes the metadata section over OCR text.

6
parseVenue(rawText)

Applies regex patterns to find venues: street addresses, "at Venue", "@ Venue", and labeled fields like venue:, location:, where:.

7
buildEventFields(results)

Assembles the final ExtractionResult with confidence scores for each extracted field.

Reading the Image

This is the function that turns image pixels into readable text. Walk through each line of real code alongside its plain English translation.

TypeScript export async function extractTextFromImage( imageUri: string ): Promise<string | null> { try { const { extractTextFromImage: extractText, isSupported, } = require('expo-text-extractor'); if (!isSupported) { return null; } const blocks = await extractText(imageUri); const joined = blocks.join('\n').trim(); return joined || null; } catch { return null; } }
Plain English
Define a function that takes an image file path and promises to return text (or nothing).
Start a safety net — if anything goes wrong, we handle it gracefully.
Load the OCR library on demand (not upfront, so the app works even without it).
Check if this device supports text recognition — not all do.
If not supported, politely give up and return nothing.
Ask the OCR engine to read all text blocks from the image.
Combine all the text blocks into one string, separated by line breaks.
Return the text if we found any, otherwise return nothing.
If anything crashed, just return nothing instead of breaking the whole app.
🛡
Graceful Degradation

This pattern — trying something and returning null instead of crashing — is called graceful degradation. It means the app keeps working even when one piece fails.

Finding the Date

Dates in event posts come in every format imaginable — "next Saturday", "March 28th at 10pm", "03/28/2026". The parseDateTime function delegates this hard problem to chrono-node.

[ String(start.get('hour')).padStart(2, '0'), String(start.get('minute') ?? 0).padStart(2, '0'), ].join(':') : null;
Plain English
Ask chrono-node to find any dates or times mentioned in the text.
If it found nothing, give up — no date, no time.
Take the first date it found (usually the most prominent one).
Pull out the specific start date/time details.
Build a clean date string like 2026-03-28 from year, month, day.
Pad single-digit months and days with a zero (3 becomes 03) using padStart.
Only include the time if chrono-node is confident it found one.
Format it as HH:MM (like 22:00 for 10 PM).
📅
The Power of chrono-node

chrono-node understands natural language dates — "next Saturday at 8pm", "March 28", "tomorrow evening". It is one of the most powerful date parsing libraries available.

Once merged, parseVenue scans the combined text using regex patterns in priority order. The most specific patterns win.

1
Street Address

123 Main St — highest priority. A numbered street address is the most reliable indicator of a physical venue.

2
"at Venue"

at The Warehouse — second priority. The word "at" before a capitalized name strongly suggests a venue.

3
"@ Venue"

@ Brooklyn Bowl — third priority. The @ symbol is informal but commonly used for venues in social media posts.

4
Labeled Field

venue: Madison Square Garden — fourth priority. Explicit labels like venue:, location:, or where: are clear but less common.

04

Saving to Calendar

How PostCal goes from extracted data to a real calendar event on your phone.

User Edits Trump Everything

The extraction pipeline is like a secretary who fills out a draft — but the boss (the user) has final say. Any edit the user makes on the review screen overrides what the AI extracted.

PostCal resolves every field through a three-tier precedence chain. The first non-empty value wins.

1
User Edit (highest priority)

Whatever the user typed on the review screen. If they changed it, that value is final.

2
Extraction Result (fallback)

What the AI pipeline pulled from the Instagram post. Used only if the user did not edit the field.

3
Default Value (last resort)

"Untitled Event" for the name, today at 7:00 PM for the time, and empty string for the location.

Code const title = userEdits.eventName ?? extraction?.eventName ?? 'Untitled Event'; const location = userEdits.venue ?? extraction?.venue ?? '';
Plain English

Use whatever the user typed for the event name.

If they did not edit it, use what the extraction pipeline found.

If extraction also found nothing, call it "Untitled Event".

Same logic for location — user edit, then extraction, then empty string.

??
What is ?? (nullish coalescing)?

The ?? operator means "if the left side is null or undefined, use the right side instead." It lets PostCal chain fallbacks cleanly in a single expression.

Asking Permission

Mobile apps cannot just write to your calendar. The operating system requires every app to request permission first — and the user can say no.

Here is exactly what happens when the user taps "Save to Calendar."

1
User taps "Save to Calendar"

The button press triggers the save flow inside PostCal.

2
App calls requestCalendarPermission()

PostCal asks the OS whether it already has calendar access, or needs to prompt the user.

3
First time: native dialog appears

The OS shows its own popup: "Allow PostCal to access your calendar?" This dialog is not drawn by PostCal — it comes from iOS or Android itself.

4
User grants permission

The OS remembers the choice and PostCal proceeds to write the event.

5
User denies permission

PostCal exits the save flow silently. No crash, no retry loop — the user simply stays on the review screen.

🔒
Why permissions exist

Your calendar contains private data — meetings, medical appointments, personal plans. The operating system enforces that apps must always ask before reading or writing to it. This is an OS-level rule, not something the app can skip.

Writing the Event

Once permission is granted, PostCal calls writeEvent to create the actual calendar entry. This function talks to the calendar API through Expo's Calendar module.

Code const defaultCalendar = await Calendar.getDefaultCalendarAsync(); let calendarId = defaultCalendar.id; const eventId = await Calendar.createEventAsync(calendarId, { title: event.title, startDate: event.startDate, endDate: event.endDate, location: event.location, timeZone: Intl.DateTimeFormat() .resolvedOptions().timeZone, }); return eventId;
Plain English

Ask the system for the default calendar (the one you probably use most).

Grab its ID so we know where to save the event.

Create a new event in that calendar with all the details.

Set the title — what the user approved on the review screen.

Set when it starts and ends (default duration: 2 hours).

Set the location if we have one.

Use the timezone from the device so times display correctly.

Return the new event ID — proof it was saved successfully.

📱
Android vs iOS

On Android, getDefaultCalendarAsync might not exist. PostCal uses a fallback: it scans all calendars and picks the first writable one. On iOS, the default calendar is always available.

⏱
Async/await in action

Every line with await pauses until the promise resolves. getDefaultCalendarAsync waits for the OS to respond; createEventAsync waits for the event to be written. If either fails, the error propagates up to PostCal's error handler.

After the event is saved, PostCal shows a success screen that auto-dismisses after 2 seconds. On devices with reduced-motion preferences enabled, the dismiss animation is replaced with a simple fade to respect accessibility settings.

When Things Break

Error handling, graceful degradation, and debugging patterns.

The Safety Net Philosophy

PostCal is built like a car with crumple zones — when something goes wrong, the damage is contained to one area. The OCR can fail without taking down the parser. The parser can fail without crashing the form.

The form always works, even if every extraction returned nothing. This philosophy of graceful degradation is baked into every layer.

🛡️
Null, Not Crash

Every parser returns null on failure, never throws. The caller simply moves on when a field is missing.

🔒
Try-Catch Everything

Native modules like OCR and calendar are wrapped in try/catch blocks so platform-level failures never propagate.

🔄
Status Machine

Clear states prevent impossible transitions. You cannot save while extracting or extract while saving.

🚦
Navigation Guards

Refs prevent duplicate saves and race conditions during screen transitions.

🪢
Graceful Fallbacks

The empty form is always available as a last resort. Even if every extraction returned nothing, the user can type manually.

💡
Think of it this way

PostCal has 94 tests across 13 test suites. But tests only catch known problems. The safety net patterns above catch the unknown ones — the blurry images, the weird date formats, the phones with broken calendar APIs.

Why a state machine?

By defining exactly which states are possible and which transitions are allowed, you prevent impossible situations — like trying to save when there is nothing to save.
