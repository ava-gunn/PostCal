04

Saving to Calendar

How PostCal goes from extracted data to a real calendar event on your phone.

User Edits Trump Everything

The extraction pipeline is like a secretary who fills out a draft — but the boss (the user) has final say. Any edit the user makes on the review screen overrides what the AI extracted.

PostCal resolves every field through a three-tier precedence chain. The first non-empty value wins.

1
User Edit (highest priority)

Whatever the user typed on the review screen. If they changed it, that value is final.

2
Extraction Result (fallback)

What the AI pipeline pulled from the Instagram post. Used only if the user did not edit the field.

3
Default Value (last resort)

"Untitled Event" for the name, today at 7:00 PM for the time, and empty string for the location.

Code const title = userEdits.eventName ?? extraction?.eventName ?? 'Untitled Event'; const location = userEdits.venue ?? extraction?.venue ?? '';
Plain English

Use whatever the user typed for the event name.

If they did not edit it, use what the extraction pipeline found.

If extraction also found nothing, call it "Untitled Event".

Same logic for location — user edit, then extraction, then empty string.

??
What is ?? (nullish coalescing)?

The ?? operator means "if the left side is null or undefined, use the right side instead." It lets PostCal chain fallbacks cleanly in a single expression.

Asking Permission

Mobile apps cannot just write to your calendar. The operating system requires every app to request permission first — and the user can say no.

Here is exactly what happens when the user taps "Save to Calendar."

1
User taps "Save to Calendar"

The button press triggers the save flow inside PostCal.

2
App calls requestCalendarPermission()

PostCal asks the OS whether it already has calendar access, or needs to prompt the user.

3
First time: native dialog appears

The OS shows its own popup: "Allow PostCal to access your calendar?" This dialog is not drawn by PostCal — it comes from iOS or Android itself.

4
User grants permission

The OS remembers the choice and PostCal proceeds to write the event.

5
User denies permission

PostCal exits the save flow silently. No crash, no retry loop — the user simply stays on the review screen.

🔒
Why permissions exist

Your calendar contains private data — meetings, medical appointments, personal plans. The operating system enforces that apps must always ask before reading or writing to it. This is an OS-level rule, not something the app can skip.

Writing the Event

Once permission is granted, PostCal calls writeEvent to create the actual calendar entry. This function talks to the calendar API through Expo's Calendar module.

Code const defaultCalendar = await Calendar.getDefaultCalendarAsync(); let calendarId = defaultCalendar.id; const eventId = await Calendar.createEventAsync(calendarId, { title: event.title, startDate: event.startDate, endDate: event.endDate, location: event.location, timeZone: Intl.DateTimeFormat() .resolvedOptions().timeZone, }); return eventId;
Plain English

Ask the system for the default calendar (the one you probably use most).

Grab its ID so we know where to save the event.

Create a new event in that calendar with all the details.

Set the title — what the user approved on the review screen.

Set when it starts and ends (default duration: 2 hours).

Set the location if we have one.

Use the timezone from the device so times display correctly.

Return the new event ID — proof it was saved successfully.

📱
Android vs iOS

On Android, getDefaultCalendarAsync might not exist. PostCal uses a fallback: it scans all calendars and picks the first writable one. On iOS, the default calendar is always available.

⏱
Async/await in action

Every line with await pauses until the promise resolves. getDefaultCalendarAsync waits for the OS to respond; createEventAsync waits for the event to be written. If either fails, the error propagates up to PostCal's error handler.

After the event is saved, PostCal shows a success screen that auto-dismisses after 2 seconds. On devices with reduced-motion preferences enabled, the dismiss animation is replaced with a simple fade to respect accessibility settings.
