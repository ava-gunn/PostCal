Meet the Cast

Every app is a team effort. Let's meet the key players that make PostCal tick.

The Cast of Characters

PostCal is made of small, focused components that each own one job. Here are the five you'll see most often.

EC
ExtractionContext — "The shared brain"

A React Context provider that holds all shared state—the shared content, extraction results, user edits, and the current status (idle → extracting → ready → saving → saved).

RS
Review Screen — "The command center"

The most complex screen. It receives share data, kicks off the extraction pipeline, displays the editable form, and handles the save action.

EP
Extraction Pipeline — "The workhorse"

Orchestrates OCR, text parsing, and result merging. Lives in lib/extraction/run-pipeline.ts and does all the heavy data lifting.

RL
Root Layout — "The stage manager"

Loads fonts, sets up the theme, wraps everything in providers, and bridges share intents into navigation events.

CM
Calendar Module — "The finish line"

The last step. Requests calendar permission, creates the event, and returns the new event ID. Lives in lib/calendar/.

⚙
Think of it like a pit crew

Each person has one job, and they work together in a precise sequence to get the car back on the track. No single component tries to do everything—they hand off cleanly.

The File Map

Here is the file tree for the PostCal project. Each folder groups files by purpose.

app/ — Screens and routing
_layout.tsx — Root layout with providers
index.tsx — Home screen (instructions)
review.tsx — Event review and edit form
success.tsx — Confirmation screen
lib/ — Business logic
extraction/ — OCR and parsing pipeline
calendar/ — Calendar integration
components/ — Reusable UI pieces
context/ — Global state (ExtractionContext)
theme/ — Design system (colors, fonts, spacing)
🔎
Notice the pattern

app/ holds screens, lib/ holds logic, context/ holds state. TypeScript files end in .tsx (UI) or .ts (logic). Keeping these apart makes changes predictable.
