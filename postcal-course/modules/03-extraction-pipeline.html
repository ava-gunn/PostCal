The Extraction Pipeline

Deep dive into how PostCal reads and understands shared content.

The Assembly Line

The extraction pipeline is like a team of translators working on a foreign document. Each specialist handles one part — dates, names, addresses — and their work gets combined into a complete translation.

No single stage does everything. Instead, seven focused steps each do one job well, then pass their results forward.

1
parseShareIntent(shareIntent)

Extracts the raw text, imageUri, and mimeType from the incoming share data.

2
extractTextFromImage(imageUri)

Runs on-device OCR via expo-text-extractor (ML Kit on Android, Apple Vision on iOS). Returns joined text blocks or null.

3
mergeExtractionSources(metadataText, ocrText)

Combines caption text and OCR text with a separator \n---\n so downstream parsers can distinguish the two sources.

4
parseDateTime(rawText)

Uses chrono-node to parse natural language dates. Returns { date: "YYYY-MM-DD", time: "HH:MM" }.

5
parseEventName(rawText)

Finds the first non-URL line of text. Prioritizes the metadata section over OCR text.

6
parseVenue(rawText)

Applies regex patterns to find venues: street addresses, "at Venue", "@ Venue", and labeled fields like venue:, location:, where:.

7
buildEventFields(results)

Assembles the final ExtractionResult with confidence scores for each extracted field.

Reading the Image

This is the function that turns image pixels into readable text. Walk through each line of real code alongside its plain English translation.

TypeScript export async function extractTextFromImage( imageUri: string ): Promise<string | null> { try { const { extractTextFromImage: extractText, isSupported, } = require('expo-text-extractor'); if (!isSupported) { return null; } const blocks = await extractText(imageUri); const joined = blocks.join('\n').trim(); return joined || null; } catch { return null; } }
Plain English
Define a function that takes an image file path and promises to return text (or nothing).
Start a safety net — if anything goes wrong, we handle it gracefully.
Load the OCR library on demand (not upfront, so the app works even without it).
Check if this device supports text recognition — not all do.
If not supported, politely give up and return nothing.
Ask the OCR engine to read all text blocks from the image.
Combine all the text blocks into one string, separated by line breaks.
Return the text if we found any, otherwise return nothing.
If anything crashed, just return nothing instead of breaking the whole app.
🛡
Graceful Degradation

This pattern — trying something and returning null instead of crashing — is called graceful degradation. It means the app keeps working even when one piece fails.

Finding the Date

Dates in event posts come in every format imaginable — "next Saturday", "March 28th at 10pm", "03/28/2026". The parseDateTime function delegates this hard problem to chrono-node.

TypeScript const results = chrono.parse(text); if (results.length === 0) return { date: null, time: null }; const ref = results[0]; const start = ref.start; const date = [ start.get('year'), String(start.get('month')).padStart(2, '0'), String(start.get('day')).padStart(2, '0'), ].join('-'); const time = start.isCertain('hour') ? [ String(start.get('hour')).padStart(2, '0'), String(start.get('minute') ?? 0).padStart(2, '0'), ].join(':') : null;
Plain English
Ask chrono-node to find any dates or times mentioned in the text.
If it found nothing, give up — no date, no time.
Take the first date it found (usually the most prominent one).
Pull out the specific start date/time details.
Build a clean date string like 2026-03-28 from year, month, day.
Pad single-digit months and days with a zero (3 becomes 03) using padStart.
Only include the time if chrono-node is confident it found one.
Format it as HH:MM (like 22:00 for 10 PM).
📅
The Power of chrono-node

chrono-node understands natural language dates — "next Saturday at 8pm", "March 28", "tomorrow evening". It is one of the most powerful date parsing libraries available.

Once merged, parseVenue scans the combined text using regex patterns in priority order. The most specific patterns win.

1
Street Address

123 Main St — highest priority. A numbered street address is the most reliable indicator of a physical venue.

2
"at Venue"

at The Warehouse — second priority. The word "at" before a capitalized name strongly suggests a venue.

3
"@ Venue"

@ Brooklyn Bowl — third priority. The @ symbol is informal but commonly used for venues in social media posts.

4
Labeled Field

venue: Madison Square Garden — fourth priority. Explicit labels like venue:, location:, or where: are clear but less common.
